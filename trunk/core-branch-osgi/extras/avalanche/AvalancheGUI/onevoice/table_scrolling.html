HTML/CSS

Each scrolling table consists of the following:

  • A div of class "headerDivOuter", within which is immediately a div of class "headerDivInner".
  • Looking further within, there are then multiple divs of class "visualColumnHeader, which each contain
    • a div of class pad0x5,
    • and then finally inside that are the column labels themselves. (So that the tables remain accessible, these column labels will be duplicated in actual th elements within the table. Read on for more details)
  • After the headerDivs there is a div of class "scrollingTableDiv", within which is the table itself, of class "scrollingTable". This div can also have additional classnames on it, of "liquidWidth", and "liquidHeight", which activate some javascript resizing behavior (described seperately further below)
  • Within the table, the first row of th elements has a class attribute of "readerCaptionRow". Display of this class will be suppressed on all screen-based browsers. (Conversely, for true aural browsers, the duplicated column header divs' display is suppressed and the readerCaptionRow element is then restored to a normal state. )
  • After the "readerCaptionRow", the first row of data cells must be of class "dummyRow", and each cell must contain a single empty div. These divs are used as shims by the javascript to wedge open the table columns when necessary to match the column header width.

Javascript

  • TableManager.js contains a number of methods to initialize scrolling when the page is loaded.
  • Most notably, a large init function whose execution is managed by the reconcileEventHandlers function within global.js, attaches all of the various event handlers required to power the component.
  • during init, an important function called resizeTableColumns will run. The purpose of this function is to reconcile the widths of the column header divs with the widths of their table columns. To represent the column side of this equation, a row of class "dummyRow" is used, whose td's each contain a single empty div. These divs will be used as shims to force the columns wider.
  • If a column header is wider than any of its data cells, the table column will be widened, using these shim divs. Conversely if the column is wider than its header, the header is widened. (Although an apparently simpler implementation might suggest itself, avoiding the dummyRow cells, and pushing the width right from the readerCaptionRow labels into the div labels, sadly there is no way to satisfactorily suppress the visual appearance of the readerCaptionRow without at the same time removing it from the tableElement's column rendering process)
  • Whenever the table is scrolled, another function called slideColumns will run. The purpose of this function is to reposition the headers left-to-right within their clipping div. (This can be seen by narrowing your browser window, and then when the horizontal scrollbars appear on the table, use them and watch the column headers.)
    Note that this function is also run from window.onresize, to workaround some specific cases where resizing can cause scrolling behaviour without triggering onscroll. This onresize handler will not interfere with any onresize handlers your application may attach, as the multiple onresize handlers will be reconciled by reconcileEventHandlers function within global.js .

Resizing tables and the LiquidWidth and LiquidHeight classes

The div of class scrollingTableDiv can accept two optional classnames, "liquidWidth", and "liquidHeight". These classes act as flags for the Javascript, turning on certain table resizing behavior.

  • The "liquidWidth" class

    Use liquidWidth if:
    1. your table is to appear within a 100% width table AND
    2. your design specifies that the ScrollingTable should grow to fill the available space in it's containing cell
    Do not use liquidWidth if:
    1. your ScrollingTable is not within a table at all, OR
    2. is within a fixed-size table, OR
    3. a table with no width specified at all
    Note: This isnt very intuitive, but if your ScrollingTable is just sitting in the page, not contained in any table or anything, it will assume all available width by default without the need for "liquidWidth". Furthermore, the use of "liquidWidth" will actually cause problems.

  • The "liquidHeight" class
    Presence of the "liquidHeight" class on a scrollingTableDiv will cause the entire table to assume all the height of it's containing cell, right up to the point where it would make that container higher. Implicit in this is the fact that some other element must be determining the height of the row, or else the scrollingTable will shrink itself to its minimum size.

    Use liquidHeight if:
    1. Your ScrollingTable (including its visualHeaderDivs) is within a tablecell, AND
    2. It is the only element within that tablecell, AND
    3. The height of the row it is in is getting determined by another element, AND
    4. you want the ScrollingTable to fill this available height.
    Do not use liquidHeight if:
    1. Your ScrollingTable is not within another table, OR
    2. Your ScrollingTable may sometimes be the only element within that table's tableRow

General Notes

  • NOTE: By design, the ScrollingTable's cells will try to use up the space available to them on the page. Resizing the page will then induce scrolling on those widths, but it will NOT reflow the table. Reloading the page at a different size however, will result in a new column layout, more appropriate to that new size. This is considered to be the best compromise, as reflowing the tables onresize would cause great problems with slower clients.
  • NOTE: The alignment of the grey header divs with the column cells is not always perfect. In some more complicated tables with very uneven contents, this alignment may end up off by a few pixels either way. Browsers implement some very complex behaviour for table columns, not all of which we were able to entirely deal with. In the majority of cases though, table headers will get aligned correctly though.
  • NOTE: in Netscape 6 and 7, as well as older versions of Mozilla (pre 1.0), ScrollingTables within 100% width tables will not resize themselves properly, although they will get laid out properly when the page loads.
