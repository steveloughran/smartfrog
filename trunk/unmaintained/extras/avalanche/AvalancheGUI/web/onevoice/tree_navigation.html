CSS

  • The CSS uses a great deal of contextual selectors to implement the spacing and interaction behaviours of the various trees, mostly through the presence or absence of certain flags on the outer div of class "TreeWrapper"
  • One note is that the Tree CSS currently supports only 5 levels of nesting. If applications require more than 5 levels of nesting, some lines will have to be added to your css.
  • Although further discussion of classnames can be found below in the HTML section, developers wishing to know more about the CSS implementation itself should refer to the css file.




HTML

  • An instance of a tree control consists of
    • On the outside, a div of class treeWrapper This div can have additional class selectors in its class attribute, seperated by space characters. The additional class selectors possible are
      • treeHasOneIconSpacing
      • treeHasTwoIconSpacing
      • treeHasCheckboxes
      • treeHasCheckboxHighlighting
      Notes:
      1. the first two flags are always mutually exclusive
      2. Navigation trees are not permitted to have the treeHasCheckboxHighlighting flag, as the highlighting behaviour that this represents is reserved for set-selection trees.
    • Within the TreeWrapper, each block of expandable or collapsible items is within a div whose class can be either "treeClosed", "treeOpen", "treeClosedSelected" or "treeOpenSelected"
    • Within this div (whether open or closed, selected or not) are three other divs.
      1. The first div is of class treeControl,
        • If the treeHasStatusIcons flag is present, each TreeControl div will contain a div of class "treeStatusIcon" which can either contain a single status icon or be empty.
        • If the treeHasCheckboxes flag is present, each TreeControl div will then contain a div of class "treeCheckbox", which can either contain a single checkbox or be empty.
        • each TreeControl div always contains an empty div of class "treeDisclosure".
      2. The second div is of class "treeTitle", and contains the text label for the tree group. In navigation trees this may be linked. This link can furthermore be associated with a highlight state change by giving the anchor tag a class of "treeSelectableLink". Note that if other components besides your Tree can make changes to the navigation state, the use of highlighting would require some page-level javascript that can push such state changes back into the Tree. For such a case, the relevant method on TreeManager is currently makeSpecificTreeNodeHighlighted(id). Another note is that although in some cases there may also be secondary links present within the tree, in no case should these be given the "treeSelectableLink" class.
      3. The third div is of class "treeContents", and contains the expandable/collapsible contents of the tree. The contents of these divs are described below
    • Within any div of class "treeContents", each row of the tree can be either
      • another div of class "treeOpen", "treeClosed", "treeOpenSelected", "treeClosedSelected", or
      • a div of class "leafWrapper", whose contents are described below.
    • Within any div of class "leafWrapper",
      • If the treeHasStatusIcons flag is present, each leafWrapper div will also contain a div of class "treeStatusIcon" which can either contain a single status icon or be empty.
      • If the treeHasCheckboxes flag is present, each leafWrapper div will also contain a div of class "treeCheckbox", which can either contain a single checkbox or be empty.
      • Each "leafWrapper" div will always contain a div of class "leaf", which contains the text label and possibly the link for the tree item

Javascript

  • TreeManager.js contains a number of methods to both initialize the trees when loaded and to manage their state at runtime.
  • Most notably, a large init function whose execution is managed by the reconcileEventHandlers function within global.js, attaches all of the various event handlers required to power the component.
  • Developers can refer to comments within the TreeManager code itself for further comments.
  • to run through the init function at a high level, when the page loads, the script picks out the treeClosed and treeOpen divs from the DOM, and attaches onclick handlers to the tree controls that they contain.
  • If the tree has the flag "treeHasCheckboxHighlighting", additional onclick handlers are attached to tie the highlighting to the checkbox state of rows.
  • Currently there is some javascript to respect other onclick handlers that developers might want to hardcode into the html. The code in these onclick handlers will be stitched in alongside the tree's own onclick handler (which is used for the expanding and contracting functionality). Combining hardcoded and dynamic event handlers like this seems to be fairly stable and it is likely that other places will be identified in this component and in others where it is a good idea to add this functionality as an extension.
  • Some methods are also made available for common application needs:
    • OpenSpecificTreeNode(id) will open all closed Tree Elements which are parents of the specified node.
    • makeSpecificTreeNodeHighlighted(id) will implement a navigation-type highlight in the given tree element
    • resizeTreeContainerTo(id,w,h) and resizeTreeContainerBy(id,dx,dy) are straightforward methods to modify the size of the tree on the page.
    • openAllTrees(id) and closeAllTrees(id) will open and close all Tree element children
    • Also, if your tree contains set-selection checkboxes, you can implement 'check all' and 'uncheck all' as follows.
      1. create a div with class "treeMasterCheckbox" containing a checkbox element.
      2. put a "mastertreeselector" attribute with a value of "yes" into the input tag.
      3. Also put a "treeid" attribute into the input tag that matches the unique id of your tree.
      When the page loads, the TreeManager will see this attach event handlers to it that automatically implement 'check all' and 'uncheck all' functionality.
